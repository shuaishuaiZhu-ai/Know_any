# Session Retrospective: 2026-04-08 18:30

## Summary

Resumed V7 CP HCQD dispatch implementation mid-task after context compaction. Primary work was fixing a literal newline byte (0x0A) that had been introduced into a C string literal in `cmd.c` during a previous session. Multiple fix attempts failed before finding a reliable approach using numeric byte arrays. Build completed clean: zero errors, zero warnings. `cmd.c` final state: 841 lines, candidate dispatch section correct.

## Errors Encountered

- **Literal newline in C string via chr() false success** — `chr(92)+chr(110)` replacement script printed "fixed ok" but the file bytes were unchanged. Root cause: the search string did not match the actual byte sequence in the file at the time the script ran (possibly a prior partial fix had already changed the surrounding bytes). No error was raised because `str.replace()` silently returns the original when the pattern is not found, and the script checked only return code, not whether a substitution actually occurred.

- **Binary mode false positive** — A second attempt using `b"\\n"` pattern also reported success. The confusion: `b"\\n"` in Python source is the two-byte sequence 0x5C 0x6E, not 0x0A. But the file still contained 0x0A at that point (or the pattern matched an intermediate state created by the first attempt). The script printed "fixed ok" without verifying the output bytes post-write.

- **Resolution** — Used `bytes([116, 32, 10, 34, ...])` → `bytes([116, 32, 92, 110, 34, ...])` constructed from the exact decimal byte values seen via `list(content[idx-1:idx+40])`. This is immune to all escaping layers: no shell processing, no Python string escape, no chr() runtime conversion — just literal integers.

## Patterns Used

- **Byte-level diagnosis before fix**: `python3 -c "f=open(path,'rb'); c=f.read(); print(list(c[idx:idx+40]))"` to read exact decimal byte values and confirm whether the target byte is 0x0A (literal newline) or 0x5C 0x6E (backslash-n escape) before writing any fix.

- **Numeric byte array replacement**: Constructed both search and replacement as `bytes([...])` with decimal integers. No string escaping at any layer. Verification after write confirmed correct bytes.

## Improvements Identified

- The existing `ssh-remote-file-editing.md` Tier 3 (`chr()`) should be extended to acknowledge that `chr()` construction in text mode can still produce a false positive if the pattern string is assembled incorrectly and `str.replace()` silently no-ops. Tier 4 should cover binary-mode `bytes([...])` with explicit post-write verification.

- Fix scripts should always verify the output file after writing, not just check whether the replace call returned without error.
