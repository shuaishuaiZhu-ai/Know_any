### 1. 总体概览

FlagScale 是一个覆盖 **大模型全生命周期** 的工具集，围绕 `flagscale` Python 包展开，提供从 **配置、训练、推理、服务、压缩、弹性容错、自动调优** 到 **多后端适配** 的完整能力。

从架构角度看，FlagScale 可以抽象为下列核心层次：
- 用户入口层：`flagscale` CLI `Flagscale/flagscale/cli.py`、`run.py`，统一入口管理训练 / 推理 / 服务任务；
- 配置与运行器层：基于 Hydra/OmegaConf 的配置系统 + `flagscale.runner.*` 运行器，实现分布式 Job 启停、脚本生成、跨节点调度与监控；
- 任务执行层：
- `train` 子系统封装 Megatron-LM / Robotics / PI0 等训练后端；
- `inference` 子系统封装 `InferenceEngine` 与 Diffusers/PI0 等推理管线；
- `serve` 子系统封装 vLLM/sglang 等在线推理引擎及“前后端解耦”的服务编排；

- 通用能力层：
- `compress` 模块进行模型/权重压缩与加速；
- `runner.auto_tuner` 和 `runner.estimator` 提供自动调优与算子/模型性能估计；
- `runner.elastic` 提供故障检测、诊断与容错机制；

- 后端与硬件适配层：`backends/`、`hardware/`、`tools/patch` 与 `train.arguments.FSTrainArguments` 中的 FlagCX/CPU 通信逻辑，完成对 Megatron-LM、vLLM 等后端以及多厂商芯片的深度适配。
下图是一个高度抽象的架构鸟瞰图：

```
<span></span>+---------------------------------------------------------------+
|                         FlagScale                             |
+---------------------------------------------------------------+
|  CLI / Entry  |  flagscale.cli  |  run.py (Hydra configs)    |
+---------------------------------------------------------------+
|  Runner       |  runner_train / runner_inference / runner_serve|
|  & Control    |  elastic monitor, auto_tuner, estimator      |
+---------------------------------------------------------------+
|  Task Logic   |  train       |  inference      |   serve     |
|               |  (Megatron)  |  (diffusers/PI0)|  (vLLM/sglang) |
+---------------------------------------------------------------+
|  Common Utils |  compress / transformations / logger / utils |
+---------------------------------------------------------------+
|  Backends &   |  Megatron-LM, vLLM, llama.cpp, robotics,     |
|  Hardware     |  FlagCX, NCCL, 多厂商芯片 & 集群拓扑          |
+---------------------------------------------------------------+

```

接下来按照 4+1 视图依次展开分析。

### 2. 逻辑视图：系统功能分解与模块关系

### 2.1 顶层功能模块划分

从逻辑上，FlagScale 主要可划分为以下功能模块：
- 配置与入口模块：
- `run.py`：统一入口脚本，通过 Hydra 加载 `examples/*/conf` 中的实验 YAML；
- `flagscale/cli.py`：提供 `flagscale train` / `flagscale serve` 等 CLI 命令，对 `run.py` 进行封装；

- 运行器（Runner）模块（flagscale/runner）：
- `runner_train.py`：分布式训练任务的调度、脚本生成、跨节点部署；
- `runner_inference.py`：批量离线推理任务调度（主要针对 vLLM 推理）；
- `runner_serve.py`：在线推理/服务的编排、端口管理、资源调度，支持分离 prefill/decoding 的解耦架构；
- `runner.elastic.*`：任务状态监控、故障检测与自动恢复；
- `runner.auto_tuner.*`：自动调参与搜索，支持异构多后端场景；
- `runner.estimator.*`：元模型（meta estimator）用于评估不同并行/算子配置的性能；**任务逻辑模块**：  - `train.*`：
- `train.arguments.FSTrainArguments`：扩展 Megatron-LM 参数，集成 FlagCX/CPU 通信与异构 process mesh；
- `train.hetero.*`：异构多维并行（tp/cp/ep/dp/pp）的进程映射与 pipeline 拆分；
- `train.dualpipev.*`：DualPipeV 调度与 overlap 策略，用于多流水线阶段的 overlapped 训练；`inference.*`：  - `inference.inference_engine.InferenceEngine`：统一的推理引擎封装，支持 Diffusers/PI0 等；
- `inference.*_entrypoint.py`：Emu3/Robobrain/Embodied 等多模态任务入口；`serve.*`：  - `serve.engine` + `serve.core` + `serve.run_*`：封装 vLLM/sglang 等推理引擎，对接 HTTP/gRPC/自定义 RPC；
- `serve.args_mapping.*`：将 FlagScale 的 config 映射到 vLLM/llama.cpp 等后端参数；

- 智能 Agent 与工具模块：
- `agent.*`：工具匹配（tool_match）、多工具协作（collaboration），用于 Embodied / 复杂工作流场景；
- `compress.*`：模型压缩、LoRA/量化等优化策略封装；
- `transformations.*`：对模型 backbone 进行 transformation，例如插入 profiling hook 或结构变换。

### 2.2 多后端与多任务类型

FlagScale 采用 **“任务类型 + 后端”** 的解耦设计：
- 任务类型（task.type）：`train`、`inference`、`serve`、`rl` 等；
- 后端（task.backend）：`megatron`、`robotics`、`pi0`、`vllm` 等；
Runner 根据 task.type 选择不同路径：
- `runner_train`：
- 解析 `train.*` 配置，构建 Megatron-LM/Robotics/PI0 的命令行参数；
- 通过 torchrun/自定义 backend 启动多进程训练；`runner_inference`：  - 解析 `inference.*` 配置为 `InferenceEngineArgs` 结构；
- 调用 `InferenceEngine` 封装的模型加载/生成/保存流程；`runner_serve`：  - 解析 `serve.*` 配置并生成 vLLM/sglang/自定义服务的配置 YAML；
- 管理多模型、多 DAG 节点服务之间的关系（包括 prefill-decoding 解耦）。
这种解耦方式使得：
- 新增后端（例如新的 MoE 引擎、国产推理框架）只需在相应子模块中实现对应 adapter；
- 上层 Runner 与 CLI 基本无需改动，极大简化了多芯片、多框架的集成成本。
### 3. 过程视图：运行时进程、线程与通信流程

### 3.1 分布式训练过程视图（以 Megatron-LM + FlagCX 为例）

训练任务由 `run.py` 或 `flagscale train` 启动，核心流程如下：

```
<span></span>用户/CLI
  |
  |  (1) python run.py --config-path=... --config-name=train
  v
Hydra 解析 YAML
  |
  |  (2) 构造 DictConfig: experiment + train + backend
  v
RunnerTrain (SSH/本地)
  |
  |  (3) _update_config_train：创建日志、ckpt 目录
  |  (4) 获取 hostfile / nnodes / nproc_per_node
  v
节点级脚本生成 (_generate_run_script_train)
  |
  |  (5) 生成 host_X_run.sh / host_X.output / host_X.pid
  v
跨节点执行脚本 (SSH / 本地 bash)
  |
  |  (6) 启动 torchrun / Megatron-LM 进程
  v
训练进程 (per GPU / per rank)
  |
  |  (7) FSTrainArguments._initialize_distributed
  |      - 设置 CUDA 设备
  |      - 初始化 torch.distributed
  |      - 若 backend == flagcx, 使用 "cpu:gloo,cuda:flagcx" 混合后端
  |  (8) RankMapper 构建异构 process mesh
  |      - 解析 hetero_process_meshes (tp, cp, ep, dp, pp)
  |      - 根据 rank 映射到不同芯片 / 并行段
  v
Megatron-LM / 自定义训练逻辑

```

进程/通信核心要点：
- torch.distributed.init_process_group：
- 默认 backend 为 NCCL / GLOO；
- 当选择 FlagCX 后端时，通过 `cpu:gloo,cuda:flagcx` 这种复合 backend 将 CPU 侧控制与 GPU/加速器侧通信分离；**异构多维并行**：  - `FSTrainArguments.pre_validate_args` 会对 `hetero_process_meshes` 做严格检查，并将其拆分为 `[tp, cp, ep, dp, pp]` 元组列表；
- 根据逻辑 rank 的范围确定当前进程所在 process mesh，并更新：
- `tensor_model_parallel_size`
- `pipeline_model_parallel_size`
- `data_parallel_size`
- `expert_model_parallel_size`**DualPipeV / overlap**：  - `train.dualpipev.*` 中实现前向/反向 overlapped pipeline（DualPipeV），在 MoE/多阶段 pipeline 训练中提升资源利用率。
### 3.2 推理过程视图（InferenceEngine）

对于离线推理任务，`runner_inference.SSHInferenceRunner` 与 `InferenceEngine` 之间的交互流程如下：

```
<span></span>用户/CLI
  |
  |  python run.py --config-path=... --config-name=inference
  v
Hydra 解析 YAML
  |
  |  (1) 构造 config.experiment + config.inference
  v
SSHInferenceRunner.__init__
  |
  |  (2) _update_config_inference: 设置 log_dir/scripts_dir/pids_dir
  |  (3) _get_args_vllm: 将 inference 配置写成 inference.yaml
  v
生成 host_run.sh
  |
  |  (4) 在脚本中设置 PYTHONPATH, 输出重定向等
  v
执行脚本 (本地/SSH)
  |
  |  (5) python <entrypoint> --config-path=inference.yaml
  v
InferenceEngine
  |
  |  (6) InferenceEngineArgs.from_yaml: 解析 engine 配置
  |  (7) create_engine_config: 组装 ModelLoadConfig + EngineConfig
  |  (8) load(): 加载 Diffusers/PI0 模型到指定 device
  |  (9) apply_transformations(): 根据 transformations cfg 修改 backbone
  | (10) generate(): 进入 RuntimeContext，会话内调用 model_or_pipeline
  | (11) save(): 将 images/video 保存到 results_path

```

该流程的关键点：
- 配置完全由 YAML 驱动，InferenceEngineArgs 利用 dataclass 反射自动解析；
- RuntimeContext 管理推理过程中需要维持的状态（如统计信息、缓存）；
- transformations 机制允许在不改动模型源码的前提下对 backbone 做插桩与结构修改。
### 3.3 在线服务过程视图（vLLM/sglang 及解耦架构）

`runner_serve` 实现了较为复杂的服务编排，包括：
- 将 Hydra 配置转换为 vLLM/sglang 的配置 YAML；
- 管理 `serve` 配置中多个服务节点（例如 router、prefill worker、decode worker、监控等）；
- 在云环境下根据 hostfile/资源自动推导 `tensor_parallel_size` 与 `pipeline_parallel_size`。
其主要流程可抽象如下：

```
<span></span>用户/CLI
  |
  |  flagscale serve <model> [yaml] --model-path=... --port=...
  v
Hydra + runner_serve
  |
  | (1) _update_config_serve
  |     - 设置 exp_dir/log_dir/scripts_dir/pids_dir
  |     - 根据 CLI 覆盖 serve.engine_args 中的 model 路径和端口
  |     - 云模式下根据 nnodes/nproc_per_node 自动设置 tensor/pipeline parallel size
  | (2) _reset_serve_port: 统一服务端口
  | (3) _get_args_vllm: 生成 serve.yaml
  v
生成 serve_run.sh
  |
  | (4) 在脚本中执行 vLLM/sglang 的启动命令
  v
服务进程
  |
  | (5) 按 serve.yaml 配置启动 model server / router / proxy
  v
客户端请求
  |
  | (6) HTTP/gRPC 请求 -> router -> model worker (vLLM/sglang)
  v
返回响应

```

在启用 **prefill-decode 解耦** 时，`runner_serve` 还会：
- 为 proxy/router 分配单独端口（`pd_proxy_port`）；
- 在 `serve` 配置中注入多个 serve 节点（prefill model、decode model 等），使得不同芯片、不同服务器可以分别承载高吞吐 prefill 和低延迟 decode 部分。

### 6. 场景视图：关键用例与交互序列

### 6.1 场景一：跨芯片异构 Megatron-LM 训练

**目标**：在含有多种 GPU/AI 芯片的集群上，对大模型进行跨芯片异构训练。

**关键参与者**：
- 用户/运维；
- `run.py` / `flagscale train`；
- `runner_train` + `FSTrainArguments` + `RankMapper`；
- Megatron-LM + FlagCX / NCCL；
**交互序列（ASCII 时序图）**：

```
<span></span>用户          run.py/CLI       runner_train         FSTrainArguments / RankMapper   Megatron-LM
 |                |                 |                         |                        |
 |  配置 YAML     |                 |                         |                        |
 |--------------->|                 |                         |                        |
 |                |  解析配置       |                         |                        |
 |                |---------------->|                         |                        |
 |                |                 |  生成 run.sh / hostfile |                        |
 |                |                 |------------------------>|                        |
 |                |                 |  启动多进程 (torchrun) |                        |
 |                |                 |------------------------>|                        |
 |                |                 |                         |  init_process_group    |
 |                |                 |                         |----------------------->|
 |                |                 |                         |  构建 process mesh     |
 |                |                 |                         |----------------------->|
 |                |                 |                         |  进入训练主循环       |
 |                |                 |                         |----------------------->|

```

在此场景下，FlagScale 通过 `FSTrainArguments` 和 `RankMapper`：
- 将 `hetero_process_meshes` 描述的多维并行配置映射到实际 rank；
- 支持异构设备类型 `hetero_device_types`，使不同厂商芯片能协同训练同一模型。
