### 概述

Gloo 是由 Facebook（现 Meta）开源的面向机器学习分布式训练的集合通信库（Collective Communications Library）。它提供了 AllReduce、Broadcast、AllGather、Barrier 等集合通信原语，支持 TCP/IP 和 InfiniBand（ibverbs）等多种网络传输，并深度集成 CUDA/NCCL 支持 GPU 场景。Gloo 是 PyTorch 分布式训练（`torch.distributed`）的核心后端之一。本文基于 Gloo v0.5.0（main 分支）进行分析。

**核心设计目标**：
- 传输无关性：算法层与传输层解耦，同一集合算法可运行在 TCP、InfiniBand、libuv 等不同传输上
- 多 GPU 支持：CUDA-aware 算法避免主机-设备内存拷贝，支持 GPUDirect RDMA
- 灵活的会合机制：通过可插拔 Store 接口（Redis、File、Hash、MPI）实现进程发现与连接建立
- 高效的环形/折半加倍算法：针对带宽受限和延迟受限场景分别优化

### 2.2 AllReduce Ring 算法数据流（N=4 个进程）

```
<span></span>初始数据各持有完整张量 T[0..3]

Rank 0: [A0|A1|A2|A3]
Rank 1: [B0|B1|B2|B3]
Rank 2: [C0|C1|C2|C3]
Rank 3: [D0|D1|D2|D3]

------ Scatter-Reduce 阶段 (N-1 轮) ------

Round 1:  0→1  (A0)    1→2  (B1)    2→3  (C2)    3→0  (D3)
          Rank0: A0+D3  Rank1: A0+B0  Rank2: B1+C1  Rank3: C2+D2

Round 2:  0→1  (A0+D3) 1→2  (A0+B0) 2→3  (B1+C1) 3→0  (C2+D2)
          Rank1: A0+B0+D3  Rank2: A0+B0+C0 ...

Round 3:  继续聚合，每个Rank获得一个完整归约后的分块

------ AllGather 阶段 (N-1 轮) ------

将归约完整的分块按环传播给所有节点，最终：

Rank 0: [ΣA|ΣB|ΣC|ΣD]   ΣX = A_x+B_x+C_x+D_x
Rank 1: [ΣA|ΣB|ΣC|ΣD]
Rank 2: [ΣA|ΣB|ΣC|ΣD]
Rank 3: [ΣA|ΣB|ΣC|ΣD]

数据流向（环形拓扑）:

  Rank0 ──send──→ Rank1 ──send──→ Rank2 ──send──→ Rank3
    ↑                                                  |
    └──────────────────recv─────────────────────────────┘

```

### 2.3 Halving-Doubling AllReduce 数据流（N=8）

```
<span></span>步骤图（8个进程，3步折半+3步加倍）

初始: 各进程持有完整数据 data[0..7]

--- Reduce-Scatter (折半阶段, log2(8)=3 步) ---

步骤1 (stride=4): 进程 {0,4} {1,5} {2,6} {3,7} 交换各自一半数据并规约
步骤2 (stride=2): 进程 {0,2} {1,3} {4,6} {5,7} 交换并规约
步骤3 (stride=1): 进程 {0,1} {2,3} {4,5} {6,7} 交换并规约
       → 每进程持有 1/8 的完整归约结果

--- AllGather (加倍阶段, 3 步) ---

步骤1 (stride=1): {0,1} {2,3} {4,5} {6,7} 互换各自分块
步骤2 (stride=2): {0,2} {1,3} 互换
步骤3 (stride=4): {0,4} {1,5} {2,6} {3,7} 互换
       → 各进程持有完整归约结果

ASCII 示意 (4进程简化版):

P0 ←─── P2          P0 ──→ P2
  ↘    ↗              ↘   ↗
   P1←─ P3              P1→ P3

step1: P0↔P2, P1↔P3    step2: P0↔P1, P2↔P3

```

### 2.4 会合（Rendezvous）过程数据流

```
<span></span>所有进程 (Rank 0..N-1)              Redis/File Store
      |                                    |
      |  1. store->set("rank_i_addr",      |
      |       my_addr_bytes)        ───────>|
      |                                    |
      |  2. store->wait(["rank_0_addr",    |
      |       ..., "rank_N-1_addr"])───────>|  (阻塞直到所有rank写入)
      |                                    |
      |  3. 读取所有地址                    |
      |     store->get("rank_j_addr")─────>|
      |                          <─────────|  (返回 addr_j)
      |                                    |
      |  4. pair->connect(addr_j) ─────────────→ Rank j TCP握手
      |     (建立与所有其他rank的TCP连接)          |
      |                                    |
      |  5. 会合完成，Context 就绪          |
      |     可执行集合通信                   |
      |                                    |

connectFullMesh 全网格连接示意（N=4）:
Rank0 ←─TCP─→ Rank1
  ↕              ↕
Rank2 ←─TCP─→ Rank3
每对 Rank 之间建立一条双向 TCP Pair

```

### 5.4 场景四：Halving-Doubling AllReduce 通信模式

```
<span></span>4 进程 Halving-Doubling (log2(4)=2步)

初始状态:
  Rank0: [a0,a1,a2,a3]  Rank1: [b0,b1,b2,b3]
  Rank2: [c0,c1,c2,c3]  Rank3: [d0,d1,d2,d3]

========= Reduce-Scatter (折半) =========

Step 1 (distance=2):
  Rank0 ←─── 发送 data[2:] ──→ Rank2
  Rank1 ←─── 发送 data[2:] ──→ Rank3

  Rank0: reduce([a0,a1], [c0,c1]) → [a0+c0, a1+c1]  (负责前半)
  Rank1: reduce([b0,b1], [d0,d1]) → [b0+d0, b1+d1]
  Rank2: reduce([a2,a3], [c2,c3]) → [a2+c2, a3+c3]  (负责后半)
  Rank3: reduce([b2,b3], [d2,d3]) → [b2+d2, b3+d3]

Step 2 (distance=1):
  Rank0 ←─→ Rank1  (各交换一半)
  Rank2 ←─→ Rank3

  Rank0: reduce → [Σ0] = a0+b0+c0+d0 (1个元素的完整归约)
  Rank1: → [Σ1]
  Rank2: → [Σ2]
  Rank3: → [Σ3]

========= AllGather (加倍) =========

Step 1 (distance=1): Rank0↔Rank1, Rank2↔Rank3
  Rank0: [Σ0, Σ1]     Rank1: [Σ0, Σ1]
  Rank2: [Σ2, Σ3]     Rank3: [Σ2, Σ3]

Step 2 (distance=2): Rank0↔Rank2, Rank1↔Rank3
  All: [Σ0, Σ1, Σ2, Σ3]  ✓ 最终结果一致

```

### 六、关键设计决策分析

### 6.1 传输层插件化设计

```
<span></span>接口与实现分离:

transport::Device (纯虚接口)
    ↓ 继承
tcp::Device      ibverbs::Device      uv::Device
    |                  |                  |
    | createContext()  | createContext()  |
    ↓                  ↓                  ↓
tcp::Context    ibverbs::Context    uv::Context
    |                  |
    | createPair()     | createPair()
    ↓                  ↓
tcp::Pair       ibverbs::Pair

算法层完全不感知底层传输，仅依赖:
  - getPair(rank) 获取 Pair 对象
  - pair->createSendBuffer() / createRecvBuffer()
  - buffer->send() / buffer->waitRecv()

```

### 6.2 Slot 机制与并发集合通信

```
<span></span>Slot 机制保证多个并发集合操作的消息隔离:

Context::nextSlot() 分配递增 slot 编号

操作 A (allreduce):  使用 slot 0, 1 (数据+通知)
操作 B (broadcast):  使用 slot 2
操作 C (barrier):    使用 slot 3

同一 Pair 上不同 slot 的消息互不干扰

```

### 6.3 UnboundBuffer recv-from-any 语义

```
<span></span>传统 Buffer（有界）:              UnboundBuffer（无界）:
  绑定到特定 Pair                  不绑定到特定 Pair
  只能从固定 rank 接收              可从任意 rank 接收 (recv-from-any)
  send/recv 严格配对               通过 slot+tag 匹配
                                   Context::Tally 仲裁竞争接收

```

### 七、总结

Gloo 通过以下核心架构决策实现了高效、灵活的分布式集合通信：

| 架构维度 | 核心设计 | 优势 |
| --- | --- | --- |
| 逻辑层 | 算法/传输/会合三层解耦 | 算法可在不同传输上复用 |
| 过程层 | epoll 异步 I/O + 用户线程 waitRecv | 低 CPU 开销，高并发 |
| 开发层 | 纯虚接口 + 模板化算法 | 易于扩展新传输和新算法 |
| 物理层 | 全网格 Pair + 多传输选择 | 适配 ETH/IB/NVLink 不同拓扑 |
| 场景层 | 可插拔 Store 会合 + 超时异常机制 | 生产环境健壮性 |

Gloo 的设计理念是”传输无关的集合算法库”，其全网格点对点连接模型、基于 epoll 的异步 I/O 以及灵活的 Store 会合机制，使其成为 PyTorch 分布式训练在 CPU 场景和跨节点场景下的首选通信后端。
