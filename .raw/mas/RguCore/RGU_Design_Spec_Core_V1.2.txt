详细设计
SubCore设计
UCore
每个UCore维护8个warp，负责如下功能：
取指令
每个warp维护一组指令缓存：IBuffer，当前设置为8条指令
若访问的PC落在IBuffer中，快速返回结果；
若访问的PC不在IBuffer中，则会触发ICache的读取(以4*128bit为单位)；
支持对ICache的预取操作
检查UR/UP的依赖
维护UR/UP状态表
若不满足执行条件，卡住
处理U域指令
控制类指令：NOP/Exit/分支/跳转/PC/Mask相关
计算类指令：UALU(共享1组单元)
下发T域指令
将非U域指令中的UR/UP替换为真实值，下发给后续通路
整体为简单顺序执行逻辑，不支持如下特性：
不支持乱序
不支持寄存器重命名
不支持多发射
不支持分支预测
WScheduler
处理T域的整体控制：指令检查=>操作数预取=>下发执行=>返回响应
IIssue
进行指令的检查和选择。
判断每条指令是否满足发射条件
维护寄存器状态表
接收Disp返回的执行单元状态
在满足条件的指令中选择优先级最高的发射
优先级 = F(blk_age, warp_id, unit, usr, rr)
按奇偶分组，选择优先级最高的两条普通指令，发给FDisp执行
选择优先级最高的一条tensor/fp64指令，发给CDisp执行
FDisp
维护细粒度指令的操作数预取+分发。
维护两份预取队列，收到指令后放入队列中。
从队列中选择操作数进行收集：
支持多个指令组合收集 – 比如指令1的操作数c有冲突/没有操作数c，则后续指令的操作数c可以收集
收集完成后标记为可发射
在两个队列中(共8条指令)中选择可发射的最优指令执行，考虑如下问题：
发射出的指令需要避免写冲突(历史指令的写/当拍指令的写)
IALU/FALU如果写延时一致，需要避免当拍写冲突
SFU/LSU延时不一致，只考虑历史冲突
统计如下信息，反馈给上级：
寄存器的在读状态
寄存器的在写状态
执行单元的状态
CDisp
维护粗粒度指令的操作数预取+分发。
收到指令后分成两路解析
分成多拍进行操作数收集
分成多拍进行执行，执行的前提是所需操作数收集完成
可细粒度分发，不一定等待所有操作数完成
同样统计寄存器读写/执行单元状态，反馈给上级
TrfCtrl
整体控制TRF的读写占用情况，避免TRF的bank冲突。
接收Disp的访存请求，进行仲裁，返回响应
下发读写动作给TRF
Tpart
执行通路包括寄存器堆和计算单元
采用多组多bank的sram实现寄存器堆
当前设计分为两组，对应奇偶warp可双发射
每组包含4个bank，平均意义上可支撑3读+1写的请求
寄存器交织分配到4个bank上：bank_id = (reg_id + warp_id)%4
采用Operand collector解决寄存器堆的bank冲突
预取完操作数之后，才能执行
寄存器堆按线程0-15/16-31分为两组
T16-31对应的读写操作固定比T0-15晚一拍
根据PU的个数分为几种情况：
PU数目=32时，PU单元和32个线程直接对应
PU数目=16时，第一拍处理T0-15，第二拍处理T16-31
PU数目<16时，重复多次b)过程
注意：对于warp间数据共享(如sfl)或者和uniform/ctrl单元的交互，需要注意前后拍的问题。
Tensor Core和FP64为多操作数指令，采用专用的取数buffer和控制通路。
Tensor Core
执行矩阵/FP64操作。

原子操作
数据通路

任务封装

返回通路：每个32bit代表一个原值

死锁分析
目标是满足如下规则：
Master：无论其他端口状态如何，必须能够接收已发出请求的响应(r/b)
Slave： 无论其他端口状态如何，只要自己的响应通道没有堵死，就能够接收新的请求
如果路上存在相同ID的请求，则任意接收的请求都必须返回响应
如果路上不存在相同ID的请求，可以允许部分请求卡住
(这里的能够接收是长时间的概念)
问题1：ACPY/TMA内部未预留大的读buffer
具体问题：如果写通道卡住，则读返回通道可能卡住
解决方案：修改TMA对SHM/GLB的访问通路，拆分为独立的R/W接口

block切换
在编程抽象上，不同的block间没有明确的先后关系，可以穿插执行。因此在block切换时，为追求效率，会在上一个warp下发指令后，尽快切入到新的warp。
block间切换时，需要额外考虑新的依赖问题：前一个block对shm的异步访问必须在后一个block对shm的访问之前完成。因此在新warp启动后，需要执行如下状态管理：
首先处于mem_stall状态，无法执行访存操作
等待对应block的shm的所有异步访问完成后(shm_cnt=0)，进入mem_ok状态
kernel切换
收到kernel结束命令后，会
等待所有warp执行完成
等待所有访存指令完成
flush所有非local memory的cache内容
invalid所有local memory的cache内容
返回响应

mbarrier
memory barrier的实现机制如下：
整体过程：初始化 -> 线程同步 -> stage0 -> stage1 -> stage2 -> …
采用最高位区分不同的stage
对应的，需要在shm/glb实现atom.arrive/atom.try_wait指令的支持，其中try_wait的目标是减少pooling load的过程，降低带宽占用。
其中，atom.try_wait在如下情况下会尽量等待：
1) 同一WARP的目标地址相同
2) 同一WARP的目标数据的高位相同
Try_wait必须注意避免死锁：
1) Global单元：在try_wait_number超过阈值后，会自动返回最久的try_wait
2) SHM单元：引入一个额外的timeout阈值。
cluster.sync
每个block的shared memory上分配一个barrier
Block启动时，初始化该地址为cluster.size()
考虑UMD下发
Cluster内所有线程查询cluster.first_block_shm_addr，然后访问该shm上的barrier
grid.sync
在global memory上分配一个barrier
在kernel启动前初始化为grid.size()
Grid内所有线程访问该barrier
数据依赖
寄存器依赖
寄存器依赖只发生在线程内部，由硬件自动维护，保证在编程上满足顺序执行的抽象：逐warp维护表格，记录正在读写的寄存器信息，根据该表格判断指令是否可以读/写寄存器。
普通指令
普通指令的延时较短，因此直接记录每条正在执行的指令的操作数情况。单条指令中每个操作数最大访问8个寄存器，因此表格中每一项采用sid+len表示，其中sid∈[0,255], len∈{1,2,4,8}。如何判断两个操作数有冲突：
区间 [sid_m, sid_m+ len_m)和[sid_n, sid_n+ len_n) 有交集
sid_m < sid_n+ len_n && sid_n < sid_m+ len_m
-len_m < sid_m - sid_n < len_n
8bit减法 + 4bit比较*2
访存指令
访存指令延时较长，因此按照寄存器的个数来记录占用情况(256*1bit)。这种记录方式下，不支持写后写的情况。

在计算核发出请求后，后续访存系统是完全乱序执行的，因此目前整体以NOC返回的bresp/rresp代表结束，不区分不同的屏障域(cta/gpu/sys)。
具体的，需要统计一些counter来记录，包含：
同步指令：
未完成的写指令条数
未完成的读指令条数
分shm/glb分别维护
异步指令：
分group维护异步指令的完成情况
*ATOM等价于1写+0/1读(返回原值时带读)
