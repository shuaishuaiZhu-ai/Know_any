1.
GPUID：硬件级资源标识
       GPUID（GPU 标识符）是最底层的硬件标识，用于唯一识别物理 GPU 设备。在支持多实例 GPU（MIG）技术的系统中，GPUID 的定义更加复杂。例如，NVIDIA A100 支持将单个物理 GPU 划分为最多 7 个独立的 GPU 实例，每个实例称为一个 MIG 片段，具有独立的计算、内存和带宽资源。
1.1
物理 GPU 与 MIG 实例的 ID 分配
      在现代 AI 集群中，GPUID 的分配机制需要同时支持传统的物理 GPU 和新兴的 MIG 技术。物理 GPU 的 ID 分配通常采用 PCI Bus ID 或 UUID 两种方式。为了确保两套编号系统的一致性，最佳实践是始终先设置 CUDA_DEVICE_ORDER=PCI_BUS_ID，让两套编号统一。最安全的方法是使用 PCI Bus ID 直接指定 GPU，完全绕过任何排序策略。
      对于支持 MIG 技术的 GPU，ID 分配机制更加复杂。NVIDIA 的 MIG 技术允许将基于 Ampere 及后续架构的 GPU（如 A100、H100）划分为独立且安全的 GPU 实例。在 Kubernetes 环境中，NVIDIA 提供了两种策略来暴露 MIG 设备：none 策略和single 策略。
1.
none 策略保持 nvidia-device-plugin 的运行方式与以往相同，插件不会区分启用或未启用 MIG 的 GPU，将枚举节点上的所有 GPU，使用nvidia.com/gpu资源类型使其可用。这种策略的优势在于兼容性好，但无法充分利用 MIG 技术的优势。
2.
single 策略旨在保持用户在 Kubernetes 中使用 GPU 的体验不变。MIG 设备使用与以前相同的nvidia.com/gpu资源类型进行枚举。然而，与该资源类型相关的属性现在映射到节点上可用的 MIG 设备，而不是完整的 GPU。这种策略能够让用户透明地使用 MIG 技术，无需修改现有应用程序。
1.2
GPU 虚拟化环境下的 ID 管理
       在 GPU 虚拟化环境中，GPUID 的管理面临新的挑战。NVIDIA vGPU 技术为虚拟化环境提供了 GPU 资源分配机制，其中vgpu-uuid是 vGPU 创建时分配的 UUID，用于唯一标识虚拟 GPU 设备。
在实际部署中，管理员需要通过以下方式管理 GPUID：
使用 **-i <GPU_ID>** 参数指定要使用的 GPU 设备，将 < GPU_ID > 替换为需要配置的 GPU 的实际 ID
使用 **-mig <INSTANCE_COUNT>** 参数配置 MIG（Multi-Instance GPU）实例数量
这种灵活的配置方式使得管理员能够根据具体需求，在物理 GPU、MIG 实例和虚拟 GPU 之间进行选择和配置。
2.
Node ID：计算节点标识与管理
2.1
传统节点与超节点架构下的 ID 设计
在 GPU 集群中，Node ID 的设计需要适应不同的硬件架构，特别是传统节点和超节点两种模式。在传统节点架构中，Node ID 代表物理服务器节点，每个节点都有唯一的标识符。
在超节点架构中，Node ID 的概念发生了重要变化。超节点（SuperNode）是将多个加速器节点通过超高速互连组成的 "超大节点"，使其在带宽与延迟特性上接近 "单一机器" 的体验。例如，华为 CloudMatrix 384 超节点通过新型高速互联总线实现 384 张卡互联成为一个超级云服务器，最高提供 300Pflops 的算力规模。
在超节点架构下，Node ID 具有以下特点：
层次化 ID 结构：第一层 ID 标识超节点本身，第二层 ID 标识超节点内的物理节点，第三层 ID 标识节点内的具体组件（如 GPU、CPU 等）。
逻辑与物理的分离：对外呈现为单一逻辑节点，但内部包含多个物理节点。这种设计使得超节点在调度和管理时可以作为一个整体处理，同时保留了内部的物理结构信息。
统一的资源视图：将多个物理节点的资源虚拟化为统一的资源池，通过统一的 Node ID 进行访问和管理。这种设计简化了资源调度和管理的复杂度。
2.2
节点状态监控与故障恢复中的 ID 应用
在 GPU 集群的运维管理中，Node ID 是节点状态监控和故障恢复的核心标识。节点控制器通过节点 ID 监控节点健康状况，在节点不可达时更新其 Ready 状态，并触发相应的 Pod 驱逐和重新调度机制。
节点状态监控系统的主要功能包括：
实时状态采集：通过节点 ID 定期采集节点的 CPU、内存、GPU 利用率、温度、功耗等关键指标。这些指标与特定的节点 ID 相关联，便于后续的分析和告警。
故障检测与定位：当系统检测到节点故障时，通过节点 ID 快速定位故障节点，并通过 BMC（基板管理控制器）进行远程诊断和控制。节点 ID 还用于关联故障日志和性能数据，帮助运维人员快速分析故障原因。
自动恢复机制：基于节点 ID 信息，系统可以实施自动化的故障恢复策略。例如，当检测到某个节点故障时，系统可以自动将该节点上的 Pod 重新调度到其他健康节点上，确保服务的连续性。
2.3
其他应用
      NODE ID是集群中计算节点的唯一标识符，在不同技术栈中具有不同的表现形式和管理方式。在 GPU 集群的特定场景下，节点代表物理节点即计算机台数，一台计算机可以有多个 GPU。节点 ID 的定义还包括 nnodes（物理节点数）、node_rank（物理节点序号）和 nproc_per_node（每个物理节点上进程数量，等价于 GPU 数量）等概念。这种精细化的定义方式确保了在多 GPU 节点环境中能够准确识别和管理每个计算单元。
     AI集群内单台物理/虚拟服务器（GPU服务器、管理节点、存储节点）的唯⼀标识，对应⼀台独⽴的硬

件设备，是集群资源调度、运维监控的基础单元。
