# Learning: HCQD Round-Robin V3 Design Patterns

## 1. Critical: Read actual code before designing
- V2 was designed from subagent summaries and got the pending semantics wrong
- The actual `cmd_entry()` for-loop uses `continue` per-HCQD, NOT a global blocking flag
- Always SSH into 192.168.80.116 and read full source before designing changes

## 2. pending_mask pattern for mixed pending + new scheduling
- Build a bitmask during the pending-check loop: `pending_mask |= BIT(hcqd_id)`
- After loop: `available = candidate_bitmask & ~pending_mask`
- This ensures pending HCQDs are NOT selected by round-robin (avoids double-peek for event cmds)
- But other HCQDs with new candidates ARE still scheduled (no global blocking)

## 3. Event CMD special behavior
- Event cmd (signal/wait) uses `ib_peek_packet()` but does NOT consume
- candidate bit stays 1 until `ib_read_packet()` + `ib_finish_packet()` when event resolves
- pending_mask filtering prevents round-robin from re-peeking a pending event HCQD

## 5. Round-Robin algorithm (unchanged across V2/V3)
- `start = (last + 1) % 8`
- `rotated = ((bitmask >> start) | (bitmask << (8 - start))) & 0xFF`
- `pos = ctz8(rotated)` using 16-entry lookup table
- `next = (start + pos) % 8`

## 6. Mermaid sequenceDiagram reserved keywords
- `LOOP`, `FOR`, `ALT`, `OPT`, `PAR`, `CRITICAL`, `BREAK`, `RECT`, `NOTE`, `END` are reserved (case-insensitive)
- Using them as participant IDs causes parse errors like: `Expecting 'ACTOR', got 'loop'`
- Fix: rename to non-reserved names like `MAIN`, `SCAN`, `SCHED`
