# 面试用工作笔记总结

> 这篇用于面试前快速复盘，把零散工作笔记组织成可以讲清楚的项目经历、技术能力和排障方法。

## 一句话定位

我主要做 GraceC CP firmware 相关工作，覆盖 CP loader/bring-up、MCQD/HCQD 队列调度、ringbuffer/IPC、SDMA/kernel command、stop/flush、`cmd_entry` 热路径优化，以及 KMD/UMD/平台协同问题定位。

## 核心技术栈

- Firmware：CP master/user firmware、RTOS、RISC-V/NX900、MMIO、interrupt、queue FSM。
- CP 架构：MCQD、HCQD、ringbuffer、doorbell、Interaction Buffer、iDMA、Event Table、stop/flush。
- Host 侧：UMD/KMD、IPC、PCIe、driver KO、fence、host/device memory。
- Debug：waveform、trace valid 位、kern.log、dmesg、UMD log、packet dump、版本对比。
- 平台：Palladium/PZ、EMU、Speed bridge、bootrom、multi CLS/PE、多 context。

## 可以重点讲的项目经历

### 1. cmd_entry 热路径优化

背景：CP firmware `cmd_entry` 需要高频轮询 HCQD candidate、pending、stop/flush，并快速 dispatch packet。

问题：波形显示拿到 candidate 后到真正 `peek` 存在多余读取；if body 布局还导致 beqz/ret 目标解析前产生 wrong-path fetch，trace 中出现 invalid 指令。

动作：

- 用波形和 trace valid 位区分真实执行和前端预取。
- 分析 `ib_get_candidate_bitmask()`、`ib_peek_packet()`、stop/flush 检查顺序。
- 提出 candidate cache、MCU id 固化、手写 goto 布局等优化。

结果：形成了可解释的热路径优化方案：多数 false path fall-through 到下一个判断，少数 handler 走 taken branch，减少 debug `-O0` 下错误路径预取浪费。

### 2. 多 MCQD/HCQD 并行调度 bring-up

背景：CP 需要支持多个 MCQD 并行，master MCU query 后 bind 到 HCQD 执行。

问题：多个 MCQD 只能 bind 到一个，或者 query/insert 地址异常。

动作：

- 检查 MCQD 地址是否加 base。
- 对照 MAS 确认 MCQD 需要 128B 连续存放。
- 使用两个 proc + 两个 MCQD 做最小复现。
- 验证 HCQD0/HCQD1 是否轮流取 cmd。

结果：确认 MCQD 地址布局和 query 机制的硬件约束，完成多队列调度路径的基本验证。

### 3. V9 SDMA copy / kernel command 调试

背景：V9 emu 下，UMD 第一个 copy kernel cmd 后，host/device memory 对比失败。

问题：失败稳定出现在 index 202；其他 emu 不复现。

动作：

- dump SDMA command packet，确认 operator 和 body 字段。
- 对比 host/device memory、size、地址、边界。
- 区分 UMD 组包、SDMA 执行、平台版本差异。

结果：把问题从“copy 错误”收敛成带稳定 index、版本差异和 packet 证据的可排查问题。

### 4. 平台 bring-up 和 PCIe/KMD 联调

背景：CP firmware 问题经常跨平台、bootrom、driver、KMD/UMD、fence 和硬件波形。

典型问题：

- 多 CP user reset 时序导致 NOC bus err。
- PCIe speed bridge t4/t5/t6 占用关系影响启动。
- DM1.4 下 CP 已执行完 cmd，但 fence 不返回。
- PZ1 KO 加载受 mutex 或 PCIe remove/rescan 流程影响。

方法：

- 使用最小配置回归：1cls1pe、内部 SDMA/kernel、老 loader、老打包工具。
- 保存 UMD log、kern.log、dmesg、波形。
- 先判断 CP 是否执行完，再查 host/KMD 是否收到完成信号。

## 面试可强调的方法论

1. 不只看代码，要把 UMD/KMD/firmware/hardware/platform 串起来。
2. 不直接相信波形地址，要结合 valid 位判断是真执行还是前端 wrong-path fetch。
3. 对 queue/ringbuffer 问题，优先检查地址、对齐、wrap、memory visibility 和中断时序。
4. 对平台问题，先做最小化回归，隔离 loader、CP user、KMD、硬件平台和测试镜像。
5. 性能优化要能落到具体 cycle/ns、汇编布局和热路径分支结构。

## 可准备的面试问答

**问：你做过最有代表性的 firmware 优化是什么？**  
答：可以讲 `cmd_entry` 热路径。它不是简单减少函数调用，而是通过 waveform、trace valid 和汇编布局发现 candidate 到 peek 前的多余路径，以及 beqz/ret 的 wrong-path fetch。最后用 candidate cache、全局化稳定字段、手写 goto 布局降低 hot loop 浪费。

**问：你怎么定位跨 UMD/KMD/firmware 的问题？**  
答：先确认 packet 是否正确，再确认 firmware 是否收到并执行，再看 KMD/fence/IPC 是否收到反馈。过程中保留 UMD log、kern.log、dmesg、packet dump 和波形，避免只从单层猜测。

**问：你如何验证多队列调度？**  
答：从 MAS 约束出发，确认 MCQD 128B 连续存放、地址加 base、global/local HCQD id 语义，再用两个 proc + 两个 MCQD 验证 HCQD0/1 轮流取 cmd。

**问：遇到平台差异怎么处理？**  
答：先固定变量，做最小配置回归；对比不同 emu、不同 DM 版本、不同 loader/KMD/UMD；确认 CP 内部执行和 host 侧观察是否一致。
