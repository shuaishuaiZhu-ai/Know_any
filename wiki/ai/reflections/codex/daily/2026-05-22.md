## 4. Prompt 质量复盘

- Goal 是否明确：ccproxy 多个子任务的局部目标明确；总目标收敛和最终验收仍不够明确。
- Context 是否足够：代码路径、repo 路径和具体问题较多，session 证据充足；但最终状态需要重新看当前 repo。
- Constraints 是否明确：多数子任务明确只读或限制修改范围；Daily Review 禁止 git/build/test 也明确。
- Done when 是否明确：ccproxy 子任务里部分 done when 不够统一，需要项目级验收清单。
- 是否要求先 plan 再改代码：部分任务使用 plan/read-only，符合风险控制。
- 是否要求验证：有测试设计和审查要求，但本次日报未复跑验证。
- 是否限制修改范围：安装脚本子任务限制较清楚。
- 是否明确 secret / remote / commit 边界：日报和归档已明确脱敏；业务任务需继续保持。

- 好的 prompt 模式：把复杂项目拆成 provider/CLI、OAuth、install、README、testing 等独立审查任务，便于并行分析。
- 坏的 prompt 模式：只要求“让 automation 可见”而未要求保持 scheduler-known ID，导致前一天出现漏跑。

```md
# Purpose

Consolidate many parallel ccproxy investigation or implementation sessions into one source-of-truth plan.

# Inputs

- Current repo path
- Session summaries
- Existing tests and docs

# Workflow

1. List all subtasks and their claimed outputs.
2. Re-read current repo files that should contain the changes.
3. Map each claim to code, tests, docs, or missing evidence.
4. Produce prioritized next actions.

# Output Format

Project convergence report with status, gaps, risks, and validation plan.

# Validation

No claim is marked complete without current-file or test evidence.

Do not output secrets. Do not commit or push without explicit approval.
```
