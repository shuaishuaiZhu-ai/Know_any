## 3. Codex 行为复盘

### 做得好的地方

- 能根据用户反馈继续追查 Daily Review 的证据来源，从“目录日期扫描”修正为 `LastWriteTime` 全量扫描。
- 在 Hermes 和 kernel test 长期会话中，仍能围绕远端 source-of-truth 继续定位问题，没有只靠本地推理。
- 在浏览器插件排查中，能读插件相关 session 和工具调用证据，区分主任务与 approval 子线程。
- 本次手动补跑能明确识别：今天没有生成日报，也没有 2026-05-21 session_index 记录。

### 做得不好的地方

- 昨天把 active automation 从 `automation` 迁到 `daily-codex-self-review`，但没有官方 registry 注册证据，导致今天调度器未触发。
- 之前把“用户可见目录存在”误当作“调度器会执行”，混淆了文件层和框架注册层。
- 自动化修复后没有等待或模拟下一次触发链路，只验证了静态配置。
- 日报补跑前仍需要人工追问，说明 automation 自检没有覆盖“是否实际送达消息”。

### 可能的根因

- Prompt 不清楚：没有要求 automation 变更后验证 scheduler registry 和下一次触发链路。
- Context 不足：当前没有官方 `automation_update` 工具可用，只能手改文件，无法证明 UI/registry 已同步。
- 任务范围过大：同时修改 prompt、ID、目录、memory、可见性和日报输出，容易把注册边界混在一起。
- 验证不足：只验证了 `automation.toml` 存在和字段正确，没有验证今天 09:00 JST 是否真的运行。
- Skill 不合适：需要专门的 automation registry check，而不是通用日报 checklist。
- AGENTS.md 信息不足：没有规定 automation ID/目录迁移必须保持调度器已知 ID。
- 构建/测试环境问题：不适用；本次不是业务 build/test。
- 人工 review 不足：昨天迁移后没有二次审查“会不会影响明天触发”。
- Secret handling 不当：本次遵守白名单读取；历史 session 中可能含敏感上下文，已省略具体值。
- 远程环境/权限问题：Hermes、kernel test 仍依赖远端权限和机器状态，证据不足，无法确认当前状态。

## 4. Prompt 质量复盘

- Goal 是否明确：日报目标明确；automation 可见性和调度触发目标之前不够明确。
- Context 是否足够：现在足够；证据包括 automation memory、active TOML、session_index、sessions、notes。
- Constraints 是否明确：已明确不读 git、不跑 build/test、不广扫 `.codex`。
- Done when 是否明确：应补充“下一次计划触发前，active ID 必须与调度器已知 ID 一致”。
- 是否要求先 plan 再改代码：本次是配置修复，不涉及业务代码。
- 是否要求验证：需要验证 active config、today note、session_index、scheduled run evidence。
- 是否限制修改范围：应限制在 automation config、automation memory、notes。
- 是否明确 secret / remote / commit 边界：已明确，但仍需白名单扫描规则。

- 好的 prompt 模式：要求“从框架层面排查”，促使检查调度器 ID、session_index、产物文件，而不是只看日报内容。
- 坏的 prompt 模式：只要求“让选项里可见”，没有同时要求“保持下一次 cron 触发不变”。
- 今天建议采用的 prompt 模板：

```text
目标：检查并修复 Daily Review automation 的真实触发链路。
上下文：不要读 git；只读 automation memory、active automation.toml、session_index.jsonl、sessions/**/*.jsonl、notes/codex-daily。
修改范围：只允许修改 automation config、automation memory、notes/codex-daily/YYYY-MM-DD.md；不改业务代码、不改 AGENTS.md、不创建 Skill。
验证要求：确认 active ID 与调度器已知 ID 一致；确认今天是否生成日报；确认 session_index 是否有当天自动化记录；写出没有消息的根因。
禁止事项：不广扫 .codex；不输出 secret；不跑 build/test/remote build；不 commit/push/delete。
done when：日报补跑完成，配置恢复到明天可触发状态，最终回复直接展示反思摘要和验证结果。
```

Skill 草案：

```md
---
name: codex-automation-registry-check
description: Use after changing Codex automation config, schedule, ID, directory, or memory. Do not use to modify business code or run build/test.
---

# Purpose

Verify that a Codex automation is not only present on disk, but also aligned with the scheduler-visible ID and able to produce the expected daily output.

# Inputs

- automation.toml
- automation memory
- session_index.jsonl
- sessions/**/*.jsonl
- expected output file path

# Workflow

1. Identify the scheduler-known automation ID.
2. Confirm exactly one active loadable automation.toml.
3. Confirm ID, name, status, rrule, timezone, cwd, and updated_at.
4. Check whether the expected run produced a session_index entry.
5. Check whether the expected output file was written.
6. If a run was missed, restore the scheduler-known ID before changing UI-facing names.

# Output Format

- Root cause
- Fix applied
- Verification table
- Remaining risk

# Validation

- Active config count is 1.
- Expected note exists.
- session_index has the expected run entry, or missing evidence is stated.

# Guardrails

- Do not broad-scan .codex.
- Do not output secrets.
- Do not delete automation history.
```
