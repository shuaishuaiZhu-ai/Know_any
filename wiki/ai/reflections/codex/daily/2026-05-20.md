### 做得好的地方

- 在 `fw` 会话中，按远端源码为准处理 IMC 启动到 `main` 的 wiki 需求，没有只靠记忆写结论。

- 在 `总结反思` 会话中，用户追问 skill 主动调用、CodeRabbit、Cloudinary、对话删除等工作流问题，Codex 能给出面向用户工作流的区分。
- 本次重跑验证主动发现并修正了 automation 的长期会话扫描缺陷。

### 做得不好的地方

- 前几次 Daily Review 只看当前 notes 目录和路径日期 session，导致真实主用户会话被漏掉。
- `kernel test on emu` 的 thread 创建在 05/13 目录但 05/19 继续更新，旧逻辑没有按 `LastWriteTime` 跨目录扫描。
- 早期解析还被同一 jsonl 中嵌入的后续 `session_meta` 误导，可能把 owning session 归错。
- 昨天多个会话依赖远端环境，但日报没有第一时间呈现远端任务、验证边界和未验证项。
- guardian/approval 子线程数量较多，如果不分类，会污染主用户任务复盘。

### 可能的根因

- Prompt 不清楚：旧 prompt 只说读取前一天 session，没要求按 `LastWriteTime` 全量扫描长期会话。
- Context 不足：当前项目目录只是日报输出目录，不是业务仓库；真实证据在全局 Codex session。
- 任务范围过大：一天内跨 automation、fw、Hermes、kernel test、MCP/语言概念多个主题。
- 验证不足：本次没有重新连远端确认 yesterday session 中的远端状态。
- Skill 不合适：Daily Review 的 session 归类和覆盖清单已经有稳定失败模式，适合沉淀 Skill。
- AGENTS.md 信息不足：磁盘上没有项目 AGENTS.md 固化这些 session 扫描规则。
- 构建/测试环境问题：证据不足，无法确认。
- 人工 review 不足：guardian/approval 存在，但日报主线审查不足。
- Secret handling 不当：session 中出现过凭据类上下文，本次已脱敏；是否历史输出完全合规，证据不足，无法确认。
- 远程环境/权限问题：Hermes、fw、kernel test 都依赖远端权限/代理/SSH 环境，风险较高。

## 4. Prompt 质量复盘

- Goal 是否明确：Daily Review 目标明确，但“昨天会话”的定位标准不够明确。
- Context 是否足够：现在足够；通过 automation memory、session_index、LastWriteTime session 清单补齐。
- Constraints 是否明确：明确，暂时不读 git，不跑 build/test，不输出敏感值。
- Done when 是否明确：应更新为“日报必须包含 session 覆盖清单摘要”。
- 是否要求先 plan 再改代码：本次不改业务代码，不适用。
- 是否要求验证：要求验证 session 可读、覆盖清单完整、日报结构完整。
- 是否限制修改范围：明确，只更新 `notes/codex-daily/YYYY-MM-DD.md` 和 automation memory。
- 是否明确 secret / remote / commit 边界：明确，敏感值脱敏，不 commit/push。

- 好的 prompt 模式：要求“主动解决遇到的问题并验证”，促使本次发现路径日期和 owning session 两个解析 bug。
- 坏的 prompt 模式：只说“前一天 session”，没有说明长期会话可能创建在更早日期目录。

```md

name: codex-daily-self-review
description: Use for Daily Codex review from local session history. Do not use to modify business code or run builds/tests.

# Purpose

Generate a daily review from all Codex sessions updated in the target day.

# Inputs

- automation memory
- recent daily notes
- session_index.jsonl
- sessions/**/*.jsonl selected by LastWriteTime

# Workflow

1. Compute target day in Asia/Tokyo.
2. Recursively scan all session jsonl files by LastWriteTime.
3. Determine owning session id from filename or first session_meta.
4. Classify sessions as main-user, automation, guardian/approval, or unknown.
5. Summarize every main-user session and keep guardian/approval separate.
6. Write the fixed 13-section daily note.

# Output Format

Daily note with session coverage table and risk table.

# Validation

- All updated sessions are counted.
- Every main-user session is named in the daily note.
- No secrets are quoted.
- Final line is fixed.

# Guardrails

- Do not read git while disabled.
- Do not run build/test/remote build.
- Do not modify business files, AGENTS.md, or Skill files.
```

## 8. 是否需要更新 AGENTS.md

- 建议新增的规则：Daily Review 必须按 `LastWriteTime` 递归扫描所有 `sessions/**/*.jsonl`，不能只按路径日期读取。

- 建议修改的规则：暂时删除 git 读取要求，直到用户重新要求 git 复盘。
- 建议删除的规则：无。
- 理由：这些都是本次实跑中复现的漏读根因。
- 风险：扫描所有长期会话会增加耗时，需要摘要化处理，避免输出大段原文或敏感信息。
