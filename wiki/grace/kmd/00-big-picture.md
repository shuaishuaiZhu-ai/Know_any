# 00 大局观：一次请求的一生

> **这章解决什么问题**：在钻进代码之前，先用大白话搞清楚一件事——当一个应用想让 Grace GPU
> 算点东西时，从「按下计算按钮」到「拿到结果」，中间这条链路上都有谁、各自干了什么、kmd 站在哪。
> 读完这章你应该能对着[全景图](<./index.md>)讲出整个故事；具体代码细节留给后面的章节。

## 为什么需要一个「内核态驱动」

GPU 是一块插在 PCIe 插槽上的硬件。应用程序跑在**用户态**，出于安全和隔离，它**不能**直接去读写
硬件寄存器、也不能随便访问物理内存——这些都是操作系统内核的特权。于是就需要一个住在**内核态**的
「中间人」，它一边能合法地操作硬件，一边给用户态开一个受控的小窗口收发请求。

这个中间人就是 **kmd**，编译出来是内核模块 `aigc.ko`。它开的那个小窗口，就是设备节点 `/dev/aigcN`。

可以把它类比成餐厅：

- **应用 / UMD**：进店的客人，知道自己要吃什么（要算什么），但不会进厨房。
- **ajthunk 用户态库**：服务员，把客人的口语需求写成标准点菜单（`Thunk_*` API → `ioctl` 请求）。
- **kmd（`aigc.ko`）**：后厨总管，唯一能进厨房（碰硬件）的人，负责备料（显存/页表）、把菜单
  贴到出菜口（命令队列）、并在菜做好时通知服务员上菜（中断/fence）。
- **Grace GPU 硬件 + fw 固件**：厨房和厨师，真正把菜做出来。

## 出场角色

| 角色 | 在哪 | 职责 |
|---|---|---|
| UMD / 应用 | 用户态 | 发起计算，准备输入数据，等待结果。 |
| ajthunk 用户态库 | 用户态 | 把意图翻译成稳定的 `Thunk_*` C API，每个调用最终是一次 `/dev/aigc` 的 `ioctl`。 |
| **kmd（`aigc.ko`）** | **内核态** | 把 GPU 注册为 PCI 设备、开 `/dev/aigcN`；管理 GPU 显存与页表；建上下文/队列；把完成事件送回用户态。 |
| Grace GPU 硬件 | 设备 | 真正执行命令的计算硬件（含命令处理器 CP）。 |
| fw 固件 | GPU 片上微控制器 | 跑在 IMC / CP master 上，kmd 通过 CP/IMC 的 IPC 环和它协作（姊妹项目 `../fw`）。 |

## 一次请求的一生（七步故事）

下面用一次最简单的「向量计算」（比如 saxpy：`y = a*x + y`）走一遍。**先记住这条主线，
每一步对应后面哪一章我都标了出来。**

1. **开门** — 应用通过 thunk 打开 `/dev/aigcN`。kmd 为这次打开分配一个「会话对象」
   `aigc_vdev`，挂在文件句柄上。今后这个 fd 上的所有请求都属于这次会话。
   （→ [02 数据结构](<./02-data-structures.md>)、[03 ioctl 与 ABI](<./03-ioctl-abi.md>)）

2. **建上下文** — 应用请求创建一个 **context**（`aigc_ctx`）。context 是一组资源的「账本」：
   它有自己的地址空间（页表、一个 VMID），后面分配的显存、建的队列都记在这个账本下。
   （→ [02 数据结构](<./02-data-structures.md>)）

3. **要显存** — 应用请求分配 GPU 显存。kmd 的显存管理器从堆里划一块物理内存，建一个
   `mem_handle` 句柄，并在这个 context 的**页表**里把「GPU 虚拟地址 → 物理地址」的映射写好。
   之后 GPU 用虚拟地址访问，硬件的翻译单元（TCU/MMU）会走页表找到真正的物理页。
   （→ [04 内存与页表](<./04-memory-and-pagetables.md>)）

4. **建队列** — 应用请求创建一个**命令队列**（engine）。队列的核心是一段**环形缓冲区**
   （CP ring）：一块显存，加上一对读/写指针（`rptr` / `wptr`）和一个**门铃**（doorbell）寄存器。
   kmd 把这段 ring 的显存映射给用户态，并把 doorbell 地址告诉它。
   （→ [05 提交、事件与中断](<./05-submission-events-interrupts.md>)）

5. **写命令 + 敲门铃** — 这是最容易误解的一步。应用（经 thunk）把要执行的命令**直接写进**那段
   映射好的 ring 显存里（组成 CP 命令包，里面指向输入/输出显存的 GPU 虚拟地址），然后把写指针
   `wptr` 写进 **doorbell** 寄存器。硬件的 CP 一看门铃响了，就从 ring 里把新命令取走开始执行。
   > ⚠️ **代码现状纠偏**：很多人以为提交命令是走内核 `ioctl(AIP_QUEUE_SUBMIT)`。但当前代码里
   > `aigc_ioctl_queue_submit()` 在拷完参数后**直接 `return -EFAULT`**（内核辅助提交路径暂时禁用）。
   > 真正在用的是上面这种 **用户态队列 + doorbell** 的直接提交方式（见 `kmd/test/kmd_test.c` 的
   > `update_queue_doorbell()`）。kmd 在这一步的职责是**建好 ring 和 doorbell**，而不是逐条命令转发。
   （→ [05 提交、事件与中断](<./05-submission-events-interrupts.md>)、[08 端到端](<./08-end-to-end-saxpy.md>)）

6. **硬件干活** — CP 解析命令包，调度计算单元执行 saxpy，把结果写回输出显存。

7. **通知完成** — 命令做完后，硬件通过两种机制告诉 CPU：一是往内存写一个 **fence/timestamp** 值
   （应用轮询或等待这个值变化），二是发一个 **MSI-X 中断**。kmd 的中断处理分上半部（快速应答）和
   下半部（在工作队列里处理事件、唤醒等待者），把完成事件送回用户态。应用于是知道：结果好了。
   （→ [05 提交、事件与中断](<./05-submission-events-interrupts.md>)）

整个故事就是：**开门 → 建账本 → 要内存 → 建队列 → 写命令敲门铃 → 硬件执行 → 通知完成**。
[08 章](<./08-end-to-end-saxpy.md>) 会把这条线落到真实的函数调用上。

## kmd 自己又分三层

上面 kmd 做的所有事，在代码里按「离硬件远近」分了三层。这是理解后面每一章的骨架：

![kmd 三层架构](../../../_attachments/grace/kmd/diagrams/02-layers.png)

- **驱动入口层**（`kmd/aigc/*.c`）：唯一能直接调用 Linux 内核 API 的一层，负责 PCI 探测、
  开字符设备、收 `ioctl`。
- **可移植核心 kmdlib**（`kmd/aigc/kmdlib/*`）：真正的业务逻辑（上下文、显存、页表、队列、调度、
  中断、fence）。它**不直接碰 Linux**，所有系统调用都走 `os_*` 包装，因此能被搬到别的环境测试。
- **Grace HAL**（`kmdlib/hal/grace/*`）：芯片专属后端，把「做什么」翻译成「碰 Grace 的哪个寄存器、
  给固件发哪条 IPC」。

> 为什么要分三层？把「Linux 特定的东西」「通用逻辑」「Grace 特定的东西」隔开，换内核版本时只动第一层，
> 换芯片时只动第三层，中间的核心逻辑可以稳定复用，也方便在没有真硬件时做主机侧仿真。
